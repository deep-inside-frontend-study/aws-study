## 11장 - 백엔드 서비스 이해하기

### 1) 백엔드란?

- **백엔드(Backend)**: 사용자 눈에 보이지 않는 서버 측 영역. 데이터 처리·저장, 비즈니스 로직 실행, 인증/인가, DB 연결, API 응답 등을 담당
- **프론트엔드 ↔ 백엔드 관계**: 프론트엔드가 사용자에게 화면을 보여주고 입력을 받으면, 백엔드가 그 요청을 받아 처리한 뒤 결과를 응답
- **백엔드를 직접 구성할 때의 부담**
  - OS 설정, 미들웨어 설치, 보안 패치, 장애 대응 등 모두 직접 처리해야 함
  - 트래픽이 늘어나면 서버 증설(스케일 아웃) 또는 사양 업그레이드(스케일 업) 필요 → 시간·비용 소모
- **AWS 백엔드 서비스의 의미**: 위 작업을 AWS가 관리형(Managed) 서비스로 제공 → 개발자는 비즈니스 로직에만 집중

### 2) AWS 백엔드 서비스 유형

- **컴퓨팅(Compute)**: EC2, Lambda, ECS, EKS - 코드를 실행하는 서버 자원
- **스토리지(Storage)**: S3, EBS, EFS - 데이터를 저장
- **데이터베이스(Database)**: RDS, DynamoDB, Aurora - 정형/비정형 데이터 관리
- **네트워킹(Networking)**: VPC, Route 53, CloudFront - 통신 경로·도메인·전송
- **부하 분산(Load Balancing)**: ELB - 트래픽을 여러 서버로 분산 ← **이번 장의 핵심**
- **보안(Security)**: IAM, WAF, Shield - 권한·방어
- **모니터링(Monitoring)**: CloudWatch - 로그·지표 수집

### 3) 부하 분산(Load Balancing)이 필요한 이유

- **단일 서버 한계**
  - 한 서버가 처리할 수 있는 동시 요청 수에는 한계가 있음
  - 트래픽이 몰리면 응답 지연 → 타임아웃 → 사용자 이탈
  - 서버가 다운되면 서비스 전체가 중단됨 (단일 장애점, SPOF)
- **부하 분산의 효과**
  - **가용성(Availability)**: 한 서버에 장애가 나도 다른 서버로 우회 → 무중단 서비스
  - **확장성(Scalability)**: 트래픽 증가 시 서버를 추가해 부하를 나눠 받음
  - **성능(Performance)**: 요청을 분산시켜 평균 응답 시간 단축
  - **유연성**: 무중단 배포(블루/그린, 카나리) 가능

### 4) 탄력적 로드 밸런서(ELB, Elastic Load Balancer)

- AWS가 제공하는 **관리형 로드 밸런싱 서비스**
- "탄력적(Elastic)" = 트래픽 양에 따라 자동으로 처리 용량을 확장/축소
- 별도의 로드 밸런서 장비 구매·운영 불필요, 사용한 만큼 과금
- 헬스 체크(Health Check)로 비정상 인스턴스를 자동 제외

**ELB 종류 4가지**

| 종류                     | 계층  | 프로토콜          | 특징                                                           |
| ------------------------ | ----- | ----------------- | -------------------------------------------------------------- |
| **ALB** (Application LB) | L7    | HTTP, HTTPS, gRPC | URL 경로/호스트 헤더 기반 라우팅, 마이크로서비스·컨테이너 친화 |
| **NLB** (Network LB)     | L4    | TCP, UDP, TLS     | 초저지연·초고성능, 고정 IP 지원, 수백만 RPS 처리               |
| **GLB** (Gateway LB)     | L3/L4 | IP 패킷           | 방화벽·IDS 같은 보안 어플라이언스 앞단에 배치                  |
| **CLB** (Classic LB)     | L4/L7 | HTTP, TCP         | 구형(레거시), 신규 사용 비권장                                 |

### 5) ALB vs NLB 비교 (핵심)

- **ALB**
  - 애플리케이션 계층 정보(URL, 헤더, 쿠키)로 라우팅 결정
  - 예: `/api/*` → API 서버 그룹, `/img/*` → 이미지 서버 그룹
  - WebSocket, HTTP/2 지원
  - SSL 종료(Termination) 처리 가능
- **NLB**
  - 패킷 수준에서 빠르게 분배 → 게임·금융 등 초저지연 서비스
  - 고정 IP / Elastic IP 할당 가능 (방화벽 화이트리스트 등록 시 유리)
  - 클라이언트 IP가 그대로 보존됨

### 6) 핵심 구성 요소

- **리스너(Listener)**
  - 클라이언트의 요청이 들어오는 **진입점**
  - 프로토콜과 포트를 지정 (예: HTTP:80, HTTPS:443)
  - 하나의 ELB에 여러 리스너를 둘 수 있음
- **규칙(Rule)** (ALB 한정)
  - 리스너에 들어온 요청을 어떤 조건에서 어디로 보낼지 정의
  - 조건: 호스트 헤더, 경로 패턴, HTTP 메서드, 쿼리 문자열 등
  - 동작: 특정 대상 그룹으로 전달, 리다이렉트, 고정 응답
- **대상 그룹(Target Group)**
  - 트래픽을 실제로 전달받는 **대상의 묶음**
  - 대상 유형: EC2 인스턴스, IP 주소, Lambda 함수, 컨테이너(ECS)
  - **헬스 체크** 설정: 일정 주기로 상태 확인 → 실패하면 라우팅 대상에서 제외
- **가용 영역(AZ) 분산**
  - 여러 AZ에 걸쳐 ELB를 배치 → AZ 단위 장애에도 서비스 지속

### 7) 부하 분산 알고리즘

- **라운드 로빈(Round Robin)**: 등록된 대상에 순서대로 균등하게 분배 (ALB 기본)
- **최소 미해결 요청(Least Outstanding Requests)**: 처리 중인 요청이 가장 적은 대상으로 우선 전달 → 처리 시간이 다른 요청에 유리
- **흐름 해시(Flow Hash)**: 출발지 IP·포트, 목적지 IP·포트, 프로토콜로 해시값을 만들어 동일 클라이언트는 동일 대상으로 전달 (NLB 기본, 세션 일관성)
- **가중치 기반(Weighted)**: 대상별로 가중치를 두어 트래픽 비율 조절 → 카나리 배포에 활용

### 8) ELB 활용 흐름 (실습 절차)

1. **대상 그룹 생성**: 대상 유형 선택(인스턴스/IP/Lambda) → 헬스 체크 설정
2. **EC2 인스턴스 등록**: 대상 그룹에 트래픽을 받을 인스턴스 추가
3. **로드 밸런서 생성**: ALB/NLB 선택 → VPC와 여러 AZ의 서브넷 지정
4. **보안 그룹 설정**: 리스너 포트(80/443) 인바운드 허용
5. **리스너 추가**: 프로토콜·포트 지정 → 기본 대상 그룹 연결
6. **(ALB만) 규칙 설정**: 경로/호스트별 라우팅 정의
7. **DNS 연결**: ELB의 DNS 이름을 Route 53에 별칭(Alias) 레코드로 등록
8. **검증**: 헬스 체크 정상 확인 후 트래픽 테스트

### 9) 주의할 점

- ELB 자체에도 비용 발생 (시간 단위 + 처리 데이터량)
- 헬스 체크 경로(`/health` 등)는 가벼운 응답으로 구성 (DB 조회 X)
- ALB는 클라이언트 IP가 헤더(`X-Forwarded-For`)에 담기므로 애플리케이션에서 별도 처리 필요
- HTTPS 사용 시 ACM(AWS Certificate Manager)에서 인증서 발급해 ELB에 연결

## 12장 - 서버 최적화용 백엔드 서비스 (EC2 Auto Scaling)

### 1) 클라우드 서버 최적화가 필요한 이유

- **트래픽은 일정하지 않다**
  - 서비스마다 피크 시간대가 존재 (예: 점심시간 배달 앱, 출근시간 교통 앱)
  - 이벤트·프로모션·외부 노출로 갑작스런 트래픽 폭증 발생
- **고정 용량 운영의 문제**
  - 피크 기준으로 서버를 구성 → 평소엔 자원 낭비(비용 ↑)
  - 평균 기준으로 구성 → 피크 시 장애·지연
- **이상적인 운영**: 트래픽이 많을 땐 서버를 늘리고, 적을 땐 줄이는 **자동 조절**
- **AWS 해법**: EC2 Auto Scaling이 이를 자동화해줌

### 2) 스케일링(Scaling) 개념 정리

- **스케일 업(Scale Up, 수직 확장)**
  - 한 서버의 사양(CPU·메모리·디스크)을 더 높은 등급으로 교체
  - 예: t3.small → t3.large
  - 장점: 구성이 단순함
  - 단점: 한계가 명확(서버 사양 상한), 교체 중 다운타임 발생 가능
- **스케일 다운(Scale Down)**: 사양을 낮춰 비용 절감 (수직 축소)
- **스케일 아웃(Scale Out, 수평 확장)**
  - 같은 사양의 서버 **개수**를 늘려 부하를 나눔
  - 예: EC2 인스턴스 2대 → 5대
  - 장점: 이론상 무제한 확장, 무중단으로 가능
  - 단점: 다중 서버 환경 설계 필요(상태 비저장, 세션 처리, 로드 밸런서 등)
- **스케일 인(Scale In)**: 서버 개수를 줄여 비용 절감 (수평 축소)
- **클라우드의 정석**: 스케일 업보다는 **스케일 아웃 + 스케일 인** 조합이 더 일반적

### 3) 아마존 EC2 오토스케일링이란?

- EC2 인스턴스의 **개수를 자동으로 조절**해주는 AWS 서비스 (수평 확장 자동화)
- 정의된 **최소(min) / 희망(desired) / 최대(max)** 개수 범위 내에서 동작
- CloudWatch 지표(CPU, 메모리, 요청 수 등)나 일정에 따라 자동으로 인스턴스를 시작/종료
- **장점**
  - 비용 최적화: 필요할 때만 자원을 사용
  - 가용성 향상: 비정상 인스턴스를 자동으로 교체
  - 운영 자동화: 사람이 새벽에 서버를 늘릴 필요 없음
  - ELB와 연동 시 트래픽 분산 + 자동 확장 = 강력한 조합

### 4) 핵심 구성 요소

- **시작 템플릿(Launch Template)**
  - 새 인스턴스를 어떻게 만들지 정의한 **설계도**
  - AMI, 인스턴스 유형, 키 페어, 보안 그룹, 사용자 데이터(User Data), IAM 역할, 스토리지 등
  - 버전 관리 가능 (v1, v2, v3 ... → 롤백·점진 배포)
  - (참고) 구버전 "시작 구성(Launch Configuration)"은 단종 예정, 시작 템플릿 권장
- **오토스케일링 그룹(Auto Scaling Group, ASG)**
  - 시작 템플릿을 기반으로 인스턴스를 관리하는 **논리적 그룹**
  - 핵심 설정값
    - **최소 용량(Minimum)**: 항상 유지할 최소 인스턴스 개수
    - **희망 용량(Desired)**: 평상시 유지하려는 인스턴스 개수
    - **최대 용량(Maximum)**: 늘어날 수 있는 상한
  - VPC·서브넷 지정 → 여러 AZ에 분산 배치 가능
  - ELB 대상 그룹과 연결 → 새 인스턴스 자동 등록
  - 헬스 체크: EC2 상태 또는 ELB 헬스 체크 기반으로 비정상 인스턴스 교체

### 5) 스케일링 정책(Scaling Policy)

- **수동 스케일링(Manual)**: 희망 용량을 직접 변경
- **동적 스케일링(Dynamic)**: 지표(CloudWatch) 기반 자동 조절
  - **대상 추적(Target Tracking)**: 가장 권장. 예) "평균 CPU 사용률 50% 유지"
  - **단계 조정(Step Scaling)**: 임계치 구간별로 다르게 조정 (예: CPU 60%↑ 1대, 80%↑ 3대)
  - **단순 조정(Simple Scaling)**: 알람 발생 시 정해진 수만큼 추가/제거 (쿨다운 필요)
- **예측 스케일링(Predictive)**: 머신러닝으로 트래픽 패턴을 학습해 **미리** 인스턴스 증설
- **예약 스케일링(Scheduled)**: 특정 시간/요일에 맞춰 조정 (예: 매주 월요일 오전 9시 +5대)

### 6) 동작 흐름 예시

1. ELB 뒤에 ASG로 묶인 EC2 인스턴스 2대 운영 중
2. 점심시간 트래픽 증가 → CPU 평균 75% 도달
3. CloudWatch 알람 → 대상 추적 정책 트리거
4. ASG가 시작 템플릿 기반으로 신규 인스턴스 2대 생성
5. 새 인스턴스가 ELB 대상 그룹에 자동 등록 → 트래픽 분산
6. 트래픽 감소 → 평균 CPU 30% 하락
7. ASG가 인스턴스를 자동 종료 (스케일 인) → 비용 절감

### 7) ASG 생성 절차 (실습 흐름)

1. **시작 템플릿 생성**
   - AMI, 인스턴스 유형, 키 페어, 보안 그룹, 사용자 데이터(부팅 시 실행할 스크립트) 정의
2. **오토스케일링 그룹 생성**
   - 시작 템플릿 선택
   - VPC·서브넷(여러 AZ) 선택
   - ELB 대상 그룹 연결 (선택)
3. **그룹 크기 설정**: min / desired / max 입력
4. **스케일링 정책 설정**: 대상 추적(예: CPU 50%) 또는 단계/예약 정책
5. **알림 설정**: SNS 통해 인스턴스 시작/종료 알림 받기
6. **태그 설정**: 자동 생성된 인스턴스에 태그 부여(비용 추적·관리 편의)
7. **검증**
   - 부하 테스트로 인스턴스 자동 증가 확인
   - 부하 제거 후 자동 감소 확인

### 8) 주의할 점

- **상태 비저장(Stateless) 설계**: 인스턴스가 언제든 추가/제거되므로 세션·파일을 로컬에 저장 X → 외부 저장소(RDS, ElastiCache, S3) 사용
- **워밍업/쿨다운(Warm-up / Cooldown)**: 새 인스턴스가 준비되기까지 시간이 필요. 이 시간을 무시하면 같은 알람으로 과도한 확장 발생
- **헬스 체크 종류 선택**: ELB 연동 시 ELB 헬스 체크를 켜야 애플리케이션 장애도 감지됨 (EC2 헬스 체크만 켜면 OS만 체크)
- **AZ 균형(Balanced)**: ASG는 여러 AZ에 인스턴스를 균등 배치하려 함 → 특정 AZ만 사용하면 가용성 저하
- **스팟 인스턴스 활용**: ASG에 스팟+온디맨드 혼합 구성 가능 → 큰 폭의 비용 절감
- **종료 정책(Termination Policy)**: 스케일 인 시 어떤 인스턴스를 먼저 종료할지 규칙(가장 오래된 시작 템플릿 사용 인스턴스 등) 설정 가능
