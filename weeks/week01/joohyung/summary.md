## 1장 AWS를 쓰려면 알아야 하는 넓고 얕은 배경지식

- AWS: 2006년 s3를 시작으로 ec2 출시
- 온프레미스, 프레미스, 하이브리드 클라우드(온프레미스 + 클라우드)
- 엣지로케이션으로 먼 거리의 리전 접근 시 캐싱이 있으면 히트 -> 없으면 캐시 서버 -> 없으면 오리진 서버

- 가용성 : 시스템 장애가 나도 정상적으로 작동할 수 있는 정도

- 비지니스 연속성 계획 : 운영에 영향을 주는 재해 장애를 방지하고 복구하는 대책

## 2장 AWS를 잘 쓰려면 알아야 하는 네트워크 지식

### 통신 프로토콜

#### IP

- ipv4, ipv6
  - 32비트 / 128비트
  - 사용자수 너무 많이 만들어져서 version6 나옴
  - 아직 호환성 문제 있음

#### IP 주소 체계

- 네트워크 주소 / 브로드캐스트 주소
  - 네트워크 주소는 첫번 째 주소(192.168.1.0), 브로드 캐스트 주소(192.168.1.255)는 마지막 주소
  - 이 주소들은 사용할 수 없음
- 클래스 (A ~ E)
  - 범위는 책 참고
  - 네트워크 규모에 따라 클래스 나누어져있음, 공인, 사설 주소 범위 다름
  - D와 E는 멀티캐스트 또는 연구 및 특수 용도

#### 서브넷팅

- 각 클래스의 네트워크를 작은 다위로 나누어 운영(서브넷팅)
- 프리픽스를 통해 어떤 IP들이 같은 서브넷에 속하는지 구분할 수 있다

### 아마존 VPC란

- 사용자 전용 가상네트워크

#### 구성요소

- 서브넷
- 인터넷 게이트웨이 (vpc와 인터넷 간)
- NAT 게이트웨이 (프라이빗 서브냇 -> VPC 외부 접근은 가능, 외부 -> 프라이빗은 불가능)
- 라우팅 테이블
- VPC 엔드포인트 : VPC 리소스들이 S3, 다이나모디비 등 외부 리소스 접근할 때 내부 네트워크를 통해 접근을 하도록

#### 서브넷

- 퍼블릭 서브넷
  - 외부에서 접근 가능한 리소스를 호스팅
- 프라이빗 서브넷
  - 서버, DB등 보안
- 직접 설정하진 않고 인터넷 게이트웨이를 통해 프라이빗 / 퍼블릭 설정

#### 인터넷 게이트웨이

- VPC는 기본적으로 격리된 환경임
- 리전 단위로 생성되는 서비스 , VPC / 서브넷에 속하지 않음

#### NAT 게이트웨이

- 프라이빗 서브넷이 외부와 통신하는데 사용
- 퍼블릭 서브넷에 위치
- 프라이빗 서브넷에서 라우팅 테이블을 통해 NAT게이트웨이 접근 -> NAT 게이트웨이에서 라우팅 테이블을 통해 인터넷에 연결

#### 라우팅 테이블

- 서브넷 연결
- 보통 퍼블릭 서브넷은 일대다, 프라이빗은 일대일 (상황에 따라 일대다도 가능)

#### VPC엔드포인트

- 인터페이스 엔드포인트
  - 사설 IP주소를 할당받아 프라이빗 링크를 통해 VPC 내의 서버가 인터넷을 거치지 않고 AWS의 다른 서비스와 통신할 수 있게해줌
  - 프라이빗 IP주소를 사용
- 게이트웨이 엔드포인트
  - 인터넷 게이트웨이 / NAT 게이트웨이 없이 S3, 다이나모S#에 연결하는 엔드포인트
  - 퍼블릭 IP가 할당

### AWS 네트워크 설계 시 고려사항

#### CIDR

- 라우팅 효율성을 향상시키는 IP 주소 할당 방법
- 서브넷 마스크를 더 간단히 표현한 것
- 아래 CIDR은 AWS에서 예약되어있음
  - 10.94.4.0
  - 10.94.4.1
  - 10.94.4.2
  - 10.94.4.3
  - 10.94.4.255

#### VPC 분할 패턴

- 단일 계정으로 분할

  1. 단일 VPC 구성 : 단일 VPC 여러 환경 시스템
  2. 환경, 시스템 별 VPC 구성: 환경, 시스템에 따라서 VPC를 나눔
  3. 시스템 별 VPC 구성 : 시스템을 각 VPC로 나눔
  4. 환경별 VPC : 각 환경을 나누어 VPC로 구성

- 환경 별 AWS 계정 분할

  - 시스템별 VPC 구성 : 환경별로 계정 분할, 시스템 별로 VPC 나누기
  - 단일 VPC 구성 : 환경별로 계정 분할, 단일 VPC에 구성

- 시스템별 AWS QNSGKF
  - 환경별 구성
  - 단일 구성
