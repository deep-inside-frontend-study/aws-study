## 01. AWS 배경지식

### AWS
- 아마존이 제공하는 클라우드 컴퓨팅 플랫폼 (인프라 환경을 구축하고 운영하는데 필요한 다양한 온라인 서비스 제공)
- 인터넷을 통해 서버, 저장소, 디비, 네트워크 등 IT 인프라를 필요한 만큼 빌려쓸 수 있는 서비스
  - AWS의 클라우드 기술을 활용하여 물리적 IT 인프라를 구축하지 X고도 안정적이고 편리한 서비스 운영 가능해짐
- **종량과금제** 를 제공
  - 필요한 만큼의 자원만 사용하고 지불할 수 있는 장점 (기업이 신속하게 인프라 확장 및 축소 가능해짐)
- AWS가 앞에 붙은 서비스 <- 유틸리티 서비스 (혼자 쓰이지 않고 다른 서비스와 함께 사용되는 특징)
  - ex. AWS 람다 : 다른 서비스에 의해 관련 작업이 자동으로 실행되는 서비스

#### 클라우드 컴퓨팅
- 직접 서버를 사서 운영하는 것이 X라, 아마존의 데이터센터에 있는 컴퓨터 자원을 인터넷으로 빌려 쓰는 것
- 주요 서비스
  - EC2 : 가상 서버
  - S3 : 파일 저장소
  - RDS : 디비
  - Lambda : 서버를 항상 켜두지 않고, 특정 이벤트가 발생할 때만 코드를 실행하는 서비스
  - CloudFront : 전 세계 곳곳에 캐시 서버를 두고, 사용자와 가장 가까운 서버에서 콘텐츠를 전달하는 서비스 (CDN)


### 온프레미스 (On-Premise)
- 자체적으로 IT 인프라를 소유하여 관리 및 운영하는 것 (회사가 직접 저장소를 관리하는 것)
- 많은 투자 비용, 시간 소요 && 유지 비용

### 클라우드
- 온프에미스의 불편한 점을 해소하고자 등장
- 서버를 가상화하여 제3자가 사용자에게 저장소를 제공하는 형태
  - 물리적 IT 리소스 설치 필요 X
  - 단기간 내에 서버 구축과 확장이 가능 (활용성 좋음)
  
### 하이브리드 클라우드 (Hybrid Cloud)
- 온프레미스 환경 + 클라우드 인프라의 이점을 결합한 최적의 인프라 구축 방법
  - 온프레미스 특징 : 데이터의 시스템 보안 및 기밀성 유지
  - 클라우드 특징 : 확장성과 유연성
- 온프레미스와 클라우드를 연결하는 별도의 네트워크망(전용선) 확보 필요


### 서버리스 (severless)
- 개발 및 운영 과정에서 서버를 관리하거나 배포할 필요 없이 원하는 기능을 구현하고 실행하는 클라우드 기반의 서비스
- 24시간 내내 서버 가동 X -> 필요한 시점에만 리소스 활용 가능 (비용 절약)
- **서버리스 처리**
  - 애플리케이션에서 발생하는 다양한 이벤트에 대응하여 사전에 작성한 코드가 실행되는 것
    - ex. 데이터베이스에 새로운 데이터 입출력 이벤트 발생시(파일 업로드 등), 자동으로 서버리스 함수 실행됨
  - 데이터베이스와의 연계를 통해 별도 서버 구축 필요 X이 데이터 저장 / 필요한 정보 가져오기 가능
- 대량의 데이터를 처리하는 배치 처리와 같은 경우 서버리스 사용 권장 XXX
  - why) AWS 람다와 같은 서버리스 함수는 최대 15분 동안만 실행가능하기 때문
  - 결론) 직접 서버 구축 방법 VS 서버리스 중 적절한 채택 필요

### AWS의 글로벌 인프라 구성

#### 1. 리전 (Region)
- 서비스가 제공되는 물리적인 국가/도시 단위의 지역
- AWS는 33개의 지역에서 서비스 제공
  - 사용자에게 물리적인 거리를 좁혀 네트워크 속도 향상
  - 다른 곳의 서비스가 중단이 되어도, 다른 리전에 있는 서비스 사용을 통해 지속적인 서비스 제공 가능
  - 한국 서울 리전의 고유 코드 : lap-northeast-2

#### 2. 가용 영역 (Availability zone)
  - AWS 리전 내에 서버, 스토리지, 네트워크 장비를 저장하는 하나 이상의 물리적인 개별 데이터센터
  - 하나의 리전 내에 여러개의 가용 영역이 물리적으로 떨어져 각각 독립적 BUT 네트워크를 통해 서로 연결
    - 자연 재해 등으로부터 영향 최소화 / 시스템 가동 중지 최소화
    - 네트워크를 통해 연결되어 사용자에게 안정적이고 신뢰 가능한 클라우드 인프라 제공

#### 3. 에지 로케이션 
- 오리진 서버의 콘텐츠를 서로 다른 지역에서 빠르게 접근할 수 있도록 만드는 캐시 서버의 모음
  - 지역마다 접근하는 데이터 사이의 거리가 다르니까 접근 최적화하기 위해서
- 사용자와 오리진 서버 사이에서 데이터를 캐시하는 역할!
  - 에지 로케이션에 캐시 있는 경우 : 시용자에게 캐시 반환
  - 에지 로케이션에 캐시 없는 경우 : 오리진에서 데이터 검색해서 에지 로케이션에 저장한 뒤에 사용자에게 반환

#### **Amazon CloudFront**
- AWS가 에지 로케이션을 사용해서 제공하는 **CDN 서비스**
  - **CDN** : 전 세계에 분산된 서버를 통해 사용자에게 가장 가까운 위치에서 콘텐츠를 빠르게 전달하는 시스템

#### 에지 로케이션 VS 캐시 서버
- **에지 로케이션**
  - 용량 문제로 자주 접근하는 데이터만을 캐시
  - 다른 캐시를 삭제하여 최적화
- **캐시 서버**
  - 더 큰 캐시를 저장할 수 있기 때문에 오리진 서버로 직접 접근하는 빈도 감소
  - 단일로 존재하기에, CDN을 위한 캐시 서버가 많이 없다면 한곳에 서비스가 집중되어 트래픽 증가 위험 가능
    - 캐시 서버를 운영하는 곳 중 한곳이 중단되면 전체 시스템이 중단되는 단일 장애점 문제 발생 가능


### 가용성 (availability)
- 리전 내 시스템 장애가 나도 정상적으로 작동할 수 있는 정도 (시스템이 사용 가능한 시간의 비율)
- 가용성이 높으면 -> 시스템 사용 가능 시간 김 / 시스템 오류 등으로 중지되는 시간 짧음
- **고가용성**
  - 시스템이나 서비스가 지속적으로 정상 운영이 되는 능력

### 신뢰성 (reliability)
- 가동 시간 동안 시스템이 예상대로 작동하고 중단되지 않는 정도 (사스탬이 고장나는 시기의 간격)

### 내결함성 (fault tolerance)
- 시스템에 장애가 발생해도 시스템 성능을 저하시키지 않고 서비스를 지속하게 하는 특성
- SLA (service level agreement) 를 충족시키기 위해 시스템 성능을 저하시키지 않고 유지하는 것

#### 가용성 VS 내결합성
- 공통점 : 시스템 장애가 발생하더라도 계속 실행할 수 있도록 구성하는 것
- 차이점
  - 가용성 : 서비스의 지속성 우선시
  - 내결합성 : 서비스 품질 우선시

### 탄력성 (Elasticity)
- 시스템이 비즈니스 상황에 따라 트래픽이나 부하에 대응하여 유연하게 자동으로 조정하는 능력
  - ex. 트래픽에 따라 서버 수 조절

#### 오토 스케일링
- AWS에서 탄력성 유지를 위해 제공하는 서비스
- CPU 사용률, 네트워크 트래픽 등 미리 정의된 조건에 따라 서버를 추가/제거

### 확장성 (Scalability)
- 비즈니스가 성장함에 따라 서버 용량이 한계에 닿을때 인프라를 화장하는 것 의미
- 서버를 확장하는 방법
  - 스케일업
    - 수직 확장 (기존 서버의 성능을 향상시키는 방법)
  - 스케일아웃
    - 수평 확장 (기존 시스템과 비슷한 스펙의 서버를 추가)


## 02. AWS 네트워크 지식

### 프로토콜 (Protocol)
- 통신하기 위해 지켜야하는 규칙

#### 통신 프로토콜
- 컴퓨터나 전자기기의 원활한 통신을 위해 교환되는 자료형시에 대해 상호 합의를 해서 사용 프로토콜을 정한 것
- IP, FTP, SMTP, HTTP, TCP, UDP ...
  - ex. 이메일 보낼때 SMTP 프로토콜 사용해서 이메일 서버에 이메일 전송 
  - ex. 홈페이지 열람 시 웹브라우저가 HTTP 프로토콜을 사용하여 웹 서버로부터 웹페이지를 요청하고 받아옴

### IP (Internet Protocol)
- 인터넷에 연결된 장치를 식별할 수 있도록 각각에 부여된 고유 주소 (특정 컴퓨터의 신원을 나타냄 / 이걸 기반으로 데이터 주소받음)

#### IPv4 / IPv6
- IP 주소의 2가지 버전
- **IPv4**
  - 첫번째 인터넷 프로토콜 / 인터넷 사용자 수 급증으로 고갈될 문제 상황 있음
- **IPv6**
  - IPv4의 고갈 문제를 해결하기 위해 등장
  - IPv4의 주소 체계를 128비트로 확장하여 제공하는 인터넷 프로토콜 주소
  - 단, 아직까지 IPv4 지원하는 곳들이 많아서 기존의 인프라와 호환성 문제 존재

**본 책에서는 현재 대다수 웹사이트 / 네트워크가 사용하는 IPv4를 IP로 지칭**

### IP의 주소체계
- 0과 1로 표기하는 이진수 32자리로 구성
  - 4개의 숫자 사이에 마침표로 구분하여 표시 (8비트씩 총 4 그룹) <- 옥텟

### 서브넷팅 (Subneting)
- 각 클래스의 주어진 IP 주소 범위를 여러 작은 네트워크로 나누는 과정
- **서브넷 (Subnet)** : 서브넷팅으로 작은 단위로 나누어진 덩이리
  - 퍼블릿 서브넷
    - 인터넷과 연결되어 있어 직접 통신이 가능한 서브넷
    - 외부에서 접근 가능한 리소스를 호스팅하는데 사용 (보안 관리 주의)
  - 프라이빗 서브넷
    - 외부에서 직접 접근 제한 (민감한 데이터 주로 저장 / 보안 유지)
    - 외부와 통신하려면 퍼블릿 서브넷을 반드시 거쳐야함

### 라우팅 (Routing)
- 어떤 경로로 데이터를 보낼지 설정하는 것
- 라우터(router)를 사용하여 다른 네트워크로 데이터 전송
- 라우팅 테이블(Routing table)을 통해 경로를 저장하고 관리 (라우터 안에 있음)

### 아마존 VPC (Virtual Private Cloud)
- 사용자 전용 가상 네트워크 / AWS의 클라우드 환경에서 네트워크를 구성하고 관리할 수 있는 서비스
  - 사용자 마음대로 가상 네트워크 환경인 VPC를 생성하는 것 (ex. 라우팅 테이블 구성, IP 주소 범위 선택, 서브넷 생성)
- VPC 1개는 **서브넷 N개를 가질 수 있음**

### VPC 엔드포인트
- VPC의 외부 서비스나 다른 서비스와 안전하게 연결하는 데 도움을 주는 서비스

#### 인터페이스 앤드포인트 (interface Endpoint)
- 사설 IP 주소를 할당받아 AWS에서 제공하는 프라이빗 연결 공간이 프라이빗링크를 통해 VPC 내의 서버가 인터넷을 거치지 않고 AWS의 다른 서비스와 안전하게 통신하게 도움
- VPC의 사설 IP, 프라이빗 IP 주소를 사용

#### 게이트웨이 앤드포인트 (Gateway Endpoint)
- 인터넷 게이트웨이와 NAT 게이트웨이 없이 아마존 S3와 아마존 다이나모DB에 연결하는 엔드포인트 유형
- 공용 IP가 할당되어 퍼블릭 IP로 접근

### AWS 네트워크 설계시 고려사항
#### CIDR 나누는 기준
- 사용할 IP 주소의 양 && 서브넷 확장 가능성 고려 필요
  - if. 서브넷 확장을 고려하고 있는 환경
    - /28로 설정하여 여유분이 있는 서브넷을 만드는 것이 권장
  - if. 서브넷 확장이 불필요하고 IP 주소 수가 중요한 경우
    - /27로 서브넷 생성 권장

### AWS 네트워크 구축을 위한 서비스
- AWS에서 네트워크 구축할때 서로 다른 VPC 간 연결 / 온프레미스 환경과 VPC 연결하고 싶을때 사용 가능한 서비스

#### VPC 피어링
- 서로 다른 VPC를 연결하여 데이터 송수신을 도와줌 (일대일 관계)
- 무료

#### Site to Site VPN
- 온프레미스와 AWS 사이의 연결
- IPsec 프로토콜을 사용하여 암호화된 터널을 설정 (안전한 연결)
- 단기간 도입 가능

#### 다이렉트 커넥트
- 온프레미스와 AWS 사이의 연결
- 물리 전용선을 통해 비공개 접속을 시도 / 고속 네트워크 대역에서 안정적인 통신을 제공
- 초기 설정 및 구축에 시간 비용 소요 가능
