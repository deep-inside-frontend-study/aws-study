# 12 ~ 14장

## 오토스케일링

### 스케일 업/아웃/인

- 스케일 업
  - 서버 성능을 향상 -> 인스턴스 사양을 업그레이드
- 스케일 아웃
  - 서버 성능을 향상 -> 같은 사양의 인스턴스를 추가
- 스케일 인
  - 스케일 아웃과 반대 -> 필요 없는 서버 삭제

### aws의 오토스케일링 / ec2의 오토스케일링

- aws
  - 여러 서비스의 스케일링 작업 (ex: db, ecs 등)
- ec2
  - 시작 템플릿으로 "오토 스케일링 그룹" 생성 -> elb랑 같이 사용 가능
  - 오토 스케일링 그룹 안에는 ec2 인스턴스"들"이 존재
  - 여러 버전을 만들어 상황에 맞게 사용 가능

### 오토 스케일링 그룹

- desired capacity (원하는 용량) -> 초기 생성할 인스턴스 수
- minimum capacity (최소 용량) -> 말 그대로
- maximum capacity (최대 용량) -> 말 그대로

### 로드 밸런서

- 개별적으로 생성해도 되고, 그룹 생성하면서 만들어도됨
  - 없이 생성도 가능 (그럼 스케일링을 누가 담당하지?)
- health / unhealthy 반환
  - unhealthy 반환하면 새로운 ec2 인스턴스로 교체
  - 로드밸런서만 있으면 제외시키는것만 가능

### 스케일 정책

- 수동으로도 가능
- 동적 크기 조정 정책, 예측 크기 조정 정책

#### 동적 크기 조정 정책

- 대상 추적 크기 조정
  - 선택한 지표 유형을 기반
  - cpu 사용률, 평균 네트워크 입/출력, 대상별로 alb 요청 수 중 하나를 선택
- 단계 크기 조정
  - 클라우드워치 경보를 기반
  - 클라우드워치라는 서비스가 있는 듯
- 단순 크기 조정
  - 오토스케일링 그룹 서비스가 처음 만들어졌을때부터 있던 정책
  - 클라우드 워치 경보 기반
  - 오타 찾음 (단게 -> 단계)
  - 스케일링으로 인스턴스 생성되고 지정된 대기 시간이 끝날때까지 기다림 -> 요즘 안씀

#### 예측 크기 조정 정책

- 머신러닝을 이용해 클라우드 워치의 기록 데이터 기반
- 시간별로 배치 설정도 가능

### 오토스케일링 그룹과 AMI (이미지)

- 인스턴스가 생성될 때, 사용자가 수동으로 설치한건 적용안됨
- 그래서 ami 기반으로 생성하면 됨

### 종료 정책

- 조건에 일치하는 인스턴스 찾아내기

  - 종료 방지 기능이 서렂ㅇ되지 않은 인스턴스
  - 가장 오래된 시작 템플릿이 있는 인스턴스
  - 결제 시간이 가장 가까운 인스턴스

- 기타등등이있다고합니다.
